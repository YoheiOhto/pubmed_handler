# AGENTS.md

このプロジェクトはSlurmおよびPBSベースの実験管理システム（template-daily-research-experiments）で運用されています。

---

## 1. オンボーディング情報 (Onboarding Information)
ゼロからプロジェクトに参加する際に知る必要がある最小限の事実です。

### 技術スタック
- **言語・パッケージ管理**: Python (==3.12.*), `uv`による環境・パッケージ管理
- **実行環境**: Slurm / PBS (Miyabi) スケジューラ搭載 of マルチノード計算クラスター（コンテナ環境: Apptainer）

### ディレクトリ構成
```
.
├── experiments/       # 実験コード（make create_exp で作成、手動作成禁止）
│   ├── latest -> ...  # 最新実験へのシンボリックリンク
│   └── {id}_{date}_{name}/
│       ├── run_slurm.sh   # Slurm/PBS ジョブ実行スクリプト
│       ├── experiment.py  # 実験実行用Pythonスクリプト
│       └── config.yml     # 実験パラメータ・upstream定義
├── outputs/           # 実験の出力（experiment.py が OUTPUT_DIR に書き出す）
├── logs/              # 実行ログ（slurm_entry.sh が自動生成）
├── lib/               # 共通ライブラリ（複数実験で共有するロジック）
├── scripts/           # Slurmから呼ばれる内部スクリプト（編集禁止）
├── templates/         # create_exp が参照するテンプレート（編集禁止）
├── tools/             # makeから呼ばれるスクリプト（編集禁止）
└── AGENTS.md          # プロジェクト固有のルール（本作）
```

### 実行・テストコマンド
- **実験の作成**: `make create_exp name=<実験名>` (ID自動採番)
- **実験の実行 (ジョブ投入)**: `runx <id>` または `runx` (最新実験を投入)
- **ジョブの確認**: `lsx` または `cdx` または `groot`
- **ジョブのキャンセル**: `cancelx <job_id>`
- **事前検証 (Preflight)**: `make preflight` または `/preflight`
- **ログの整理**: `make log_clean`
- **失敗・Orphanedログの削除**: `make clean_failed`

## 2. コードベースから推論できないルール (Non-inferrable Rules)
既存コードのパターンから自然に学習できない、運用上および設計上の強い制約です。

### 実験の粒度: 1実験 = 1フェーズ
複数フェーズ（例：前処理とモデル学習など）を1つの実験ディレクトリに詰め込むことは禁止です。

| フェーズ | 実験名の例 | 主な出力 |
|---------|-----------|---------|
| データ取得・ES検索 | `fetch_patients` | `retrieved.jsonl` |
| 前処理・特徴量生成 | `preprocess_sequences` | `features.parquet` |
| モデル学習 | `train_mamba_ehr` | `checkpoint.pt` |
| 評価・推論 | `eval_mamba_ehr` | `results.parquet` |
| 結果解析・集計 | `analyze_results` | `figures/` |

### 実行の粒度: 1ジョブ = 1原子単位
- `experiment.py` 内で可変次元（モデル、シード、ハイパーパラメータ等）をループで回して複数実行してはいけません。
- 1つのジョブが処理するのは、可変次元の**1つの組み合わせのみ**です。
- 複数組み合わせの実行は、必ず外部から引数で渡すように設計し、`runx --array` (GRID_ARGS と GRID_VALUES の直積) または `runx --seq` を使用して並列/順次投入してください。
- 探索する可変次元は `run_slurm.sh` の `--arg` 経由で受け取り、固定パラメータは `config.yml` に記載します。

### 時間の余裕（TIMEOUTは全損）
- ジョブに指定する `--time` や `DEFAULT_TIME` は、想定される実行時間の **1.5〜2倍** の余裕を持たせてください。TIMEOUTすると結果が completed にならず、再実行が必要になります。
- 1サンプルの所要時間 × 件数 + マージン の見積もり根拠を計画書 `plan.md` に必ず記載してください。

### データの永続化と依存
- 原則として、前処理データ、実験結果、集計結果などのすべての永続化データは **Parquet 形式 (`.parquet`)** で保存・読み込みを行ってください。
- `udata/` ディレクトリからデータを取得または参照する場合は、必ず `plan.md` の `## upstream` および `config.yml` の `upstream:` にそのパスを明記してください。

### lib/ 変更時の影響範囲確認
- `lib/` を編集・追加する場合は、必ず `grep` 等で影響を受ける他の実験をすべて列挙した上で、**単独でコミット**してください。

## 3. 明確な禁止事項 (Explicit Prohibitions)
解釈の余地がなく、ツールまたはフックで制限・ブロックされる絶対的な禁止事項です。

- **特定ディレクトリの直接編集禁止**:
  - `scripts/`, `tools/`, `templates/` ディレクトリ以下のファイルは編集禁止です。
- **データ・出力への直接書き込み禁止**:
  - `outputs/` や `data/` に対する直接のコピーや変更は禁止です。出力は必ず `experiment.py` 経由で `$OUTPUT_DIR` に書き出してください。
- **for/while ループによるコマンド一括実行禁止**:
  - シェル上で `for x in ...; do <cmd>; done` 形式でまとめてジョブを投げることは禁止です。
- **Pythonファイルの1ファイル200行制限**:
  - `experiments/` または `lib/` 以下に配置する Python コードは、**1ファイルあたり200行以内**に収める必要があります。200行を超える編集・新規作成はフックによりブロックされます。共通ロジックは `lib/` に切り出しモジュール化してください。
- **mainブランチの保護**:
  - `main` / `master` ブランチへの直接の編集・コミット・チェックアウトは禁止です。
  - 作業は `work/xxx` ブランチで行い、完了時は `gfinish` で `machine/<hostname>` に集約する。
  - `gfinish` 完了後、ユーザーに `gpush` の実行を依頼して終了すること。
- **新規ブランチの作成禁止**:
  - `git checkout -b` / `git switch -c` は使用禁止です。ブランチ作成はユーザーが `gstart` で行います。
- **ブランチの強制削除禁止**:
  - `git branch -D` はユーザーに委ねてください。
- **git merge の制限**:
  - `machine/*` ブランチ上での merge（`gfinish` が使う）のみ許可されます。それ以外は禁止です。

## 4. エスカレーションのルール (Escalation Rules)
判断が必要な場面でどう振る舞うかという行動指針です。

### ユーザー承認を必ず取る場面
以下の操作は、エージェントが自律的に実行してはいけません。必ず実行するコマンドや内容をチャット上で提示し、**ユーザーの明確な承認（Proceedなど）を得てから**進めてください。
1. **ジョブの投入・キャンセル**: `runx`, `sbatch`, `srun`, `cancelx`
2. **環境設定・パッケージ追加**: `make setup`, `make uv_sync`, `uv add`
3. **Git操作**: `git push`, `git rebase` の提案（ユーザーが行う）。`gstart` / `gpush` の実行依頼もここに含む
4. **実験の複数フェーズ跨ぎ**: 1実験に複数フェーズを含める必要が生じた場合や、実験計画を大きく変更する場合
5. **破壊的操作**: `outputs/` 以下の個別ディレクトリ削除や `make log_clean` / `make clean_failed` によるログクリア

### エラー修正ループの上限（5回ルール）
- ローカル環境で実行可能な修正と再実行のループは **最大5回** までとします。
- 5回以内でエラーが解決しない場合は、強行せず、**これまでの試行履歴（発生したエラーと試行した修正内容）を整理してユーザーに報告し、指示を仰いでください。**
